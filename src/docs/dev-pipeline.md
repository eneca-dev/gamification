# Пайплайн разработки с нейросетями

Руководство для разработчика и AI-ассистента по совместной разработке функционала.

---

## Выбор пайплайна

| Масштаб | Триггер | Пайплайн |
|---------|---------|----------|
| **Мелкий** | Фикс бага, 1-2 файла, < 50 строк | [Quick Pipeline](#quick-pipeline-для-мелкихсредних-изменений) |
| **Средний** | Новый компонент, хук, 3-5 файлов | [Quick Pipeline](#quick-pipeline-для-мелкихсредних-изменений) |
| **Крупный** | Новый модуль, > 5 файлов, новые таблицы БД | [Full Pipeline](#full-pipeline-для-крупных-фич) |

# Quick Pipeline (для мелких/средних изменений)

Укороченный пайплайн для изменений, не требующих полной декомпозиции.

## Когда использовать

- Фикс бага (1-3 файла)
- Добавление нового компонента в существующий модуль
- Новый хук/action в существующей структуре
- Рефакторинг без изменения архитектуры
- UI изменения (стили, layout)

## Шаги Quick Pipeline

### Q1. Краткий анализ

```
📍 QUICK PIPELINE: Анализ задачи

**Задача:** [краткое описание]
**Масштаб:** Мелкий / Средний
**Затрагиваемые файлы:** ~N файлов

Начинаю реализацию.
```

### Q2. Реализация + Проверка агентами

Нейросеть:
1. Пишет код
2. После написания **автоматически** вызывает агентов:
   - `🤖 Cache Guardian` — если код содержит actions/hooks/data fetching
   - `🤖 Clean Code Guardian` — если создан компонент > 50 строк
3. Исправляет замечания агентов

```
📍 QUICK PIPELINE: Реализация

[Код...]

🤖 Агенты:
- Cache Guardian: ✅ Пройдено / ⚠️ Исправлено
- Clean Code Guardian: ✅ Пройдено / ⚠️ Исправлено

Запусти `npm run build` для проверки.
```

### Q3. Проверка и коммит

Разработчик:
1. Проверяет `npm run build`
2. Тестирует функционал
3. Делает коммит

```
📍 QUICK PIPELINE: Готово

Изменения готовы. Сделай коммит:
`git add . && git commit -m "тип(модуль): описание"`
```

---

## Визуальная схема Quick Pipeline

```
┌─────────────────┐
│   Q1. Анализ    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐

         ▼

│    + Коммит     │

```

# Full Pipeline (для крупных фич)

Полный пайплайн для разработки новых модулей и крупного функционала.

---

## Фаза 0: Подготовка

### Для разработчика:
1. Создать ветку под фичу: `git checkout -b feature/название-фичи`
2. Сформулировать глобальную задачу в 2-3 предложениях
3. Определить критерии готовности (Definition of Done)

### Для нейросети:
1. Изучить контекст проекта:
   - `CLAUDE.md` — основные правила и паттерны
   - `src/lib/types.ts` — общие типы проекта
   - `src/modules/<name>/types.ts` — типы модуля (если работа в существующем модуле)
   - `src/docs/<module>.md` — документация модуля (если существует)
   - Примеры существующих модулей (структура, components, actions, queries)
2. Понять текущую архитектуру и не предлагать решения, противоречащие ей
3. Проверить, нужен ли модуль `@/modules/cache`:
   - Используется, если компоненту нужны данные на клиенте (интерактивность)
   - Используется для оптимистичных обновлений и realtime-синхронизации с БД
   - Прямые импорты из `@tanstack/react-query` запрещены — только фабрики из `@/modules/cache`
   - Документация: `src/modules/cache/README.md`

## Фаза 1: Декомпозиция задачи

### Разработчик:
- Описывает глобальную цель и бизнес-требования
- Указывает ограничения и приоритеты

### Нейросеть:
- Разбивает задачу на этапы с чёткими границами
- Определяет зависимости между этапами
- Создаёт документ-план в `src/docs-features/<module>/название-фичи.md`

### 🤖 Вызов Pragmatic Architect

После создания плана **автоматически** вызывается агент:

```
📍 ПАЙПЛАЙН: Фаза 1 — Декомпозиция

[План этапов...]

🤖 Pragmatic Architect: Проверка плана на over-engineering...

⚖️ Результат:
- Необходимость: ✅
- Простота: ✅ / ⚠️ [замечания]
- Существующие решения: ✅ / ⚠️ [замечания]

Подтверди план или внеси корректировки.
```

**Формат документа-плана:**
```markdown
# Название фичи

## Цель
[Краткое описание]

## Этапы реализации

### Этап 1: [Название]
- Описание
- Затрагиваемые файлы
- Зависимости

### Этап 2: [Название]
...

## Критерии готовности
- [ ] Критерий 1
- [ ] Критерий 2
```

## Фаза 2: Цикл разработки этапа

Повторяется для каждого этапа из плана.

### Шаг 2.1: Планирование этапа

**Разработчик:**
- Уточняет требования к этапу
- Задаёт вопросы по неясным моментам

**Нейросеть:**
- Составляет детальный план реализации:
  - Архитектура решения
  - Используемые технологии и паттерны проекта
  - Новые файлы и изменения в существующих
  - Потенциальные проблемы и edge-cases
  - Интеграция с существующим кодом

### 🤖 Вызов Pragmatic Architect (для сложных этапов)

Если этап включает архитектурные решения:

```
📍 ПАЙПЛАЙН: Фаза 2.1 — Планирование этапа N

[План реализации...]

🤖 Pragmatic Architect: Проверка решения...

⚖️ Вердикт: Approved / Needs Simplification
[Замечания если есть]
```

### Шаг 2.2: Ревью плана

**Разработчик:**
- Проверяет соответствие паттернам проекта
- Оценивает сложность и риски
- Вносит корректировки
- Даёт "добро" на реализацию

**Критерии хорошего плана:**
- Нет over-engineering
- Использует существующие абстракции проекта
- Минимальные изменения для достижения цели
- Учитывает типобезопасность

### Шаг 2.3: Реализация

**Нейросеть:**
- Пошаговая имплементация согласно плану
- Проверка сборки → `npm run build`
- Проверка линтера → `npm run lint`

### 🤖 Автоматический вызов агентов после реализации

После написания кода **обязательно** вызываются агенты:

```
📍 ПАЙПЛАЙН: Фаза 2.3 — Реализация

[Код...]

🤖 Проверка агентами:

**Cache Guardian:** (для actions/hooks с data fetching)
- Server Actions: ✅ Правильный возвращаемый тип
- Валидация через Zod + safeParse: ✅
- revalidatePath/revalidateTag после мутации: ✅

**Clean Code Guardian:** (для компонентов > 50 строк)
- Структура: ✅
- Naming: ✅
- TypeScript: ✅

Запусти `npm run build` и проверь.
```

**Разработчик:**
- Следит за процессом
- Останавливает при отклонении от плана

### Шаг 2.4: Ручное тестирование

**Нейросеть составляет чек-лист:**
```markdown
## Чек-лист тестирования этапа N

### Позитивные сценарии
- [ ] Сценарий 1: [описание] → ожидаемый результат
- [ ] Сценарий 2: [описание] → ожидаемый результат

### Негативные сценарии
- [ ] Невалидные данные → корректная ошибка
- [ ] Отсутствие прав → редирект/сообщение

### Edge-cases
- [ ] Пустые данные
- [ ] Большие объёмы данных
- [ ] Параллельные операции
```

**Разработчик:**
- Проходит чек-лист вручную
- Фиксирует найденные баги
- Сообщает результаты нейросети

### Шаг 2.5: Code Review

**Нейросеть проводит ревью по критериям из `src/docs/code-review.md`:**

1. **Архитектура**
   - Соответствие паттернам проекта
   - Правильное разделение ответственности
   - Корректное использование module boundaries

2. **Дублирование кода**
   - Нет копипасты запросов к БД
   - Общие компоненты вынесены в `src/components/`
   - Переиспользование существующих хуков и утилит

3. **Безопасность**
   - Валидация входных данных через Zod + safeParse
   - Нет утечки чувствительных данных
   - Секреты только в `.env.local`, не в коде

4. **Обработка ошибок**
   - Server Actions возвращают `{ success: true, data } | { success: false, error }`
   - Ошибки отображаются пользователю, не проглатываются молча

5. **TypeScript**
   - Нет `any` нигде
   - Props через `interface`, не `type` (если нет union)
   - Типы из `src/lib/types.ts` или `src/modules/<name>/types.ts`

6. **Производительность**
   - Нет лишних клиентских запросов — данные через Server Components
   - `'use client'` поставлен как можно глубже в дереве

### Шаг 2.6: Исправления

**Нейросеть:**
- Исправляет замечания из ревью
- Объясняет каждое изменение

**Разработчик:**
- Подтверждает исправления

### Шаг 2.7: Коммит

**Формат коммита:**
```
feat(module-name): краткое описание

- Детали изменения 1
- Детали изменения 2

Этап N из M для #issue-number
```

**Один этап = один коммит** (или несколько логически связанных)

### Шаг 2.8: Переход к следующему этапу

→ Возврат к шагу 2.1 для следующего этапа

---

## Фаза 3: Завершение фичи

### Интеграционное тестирование
- Проверить взаимодействие всех этапов
- Тестирование полного user flow
- Проверка на разных ролях пользователей

### Финальный ревью

**Нейросеть проверяет:**
- Консистентность кода между этапами
- Отсутствие "мёртвого" кода
- Актуальность документации модуля — `src/docs/<module>.md`
- Обновлены ли все затронутые модульные docs

### 🤖 Финальная проверка всеми агентами

```
📍 ПАЙПЛАЙН: Фаза 3 — Финальная проверка

🤖 Pragmatic Architect:
- Общая архитектура: ✅
- Нет over-engineering: ✅

🤖 Cache Guardian:
- Server Actions соответствуют паттерну: ✅
- Валидация через Zod: ✅
- Ревалидация кэша после мутаций: ✅

🤖 Clean Code Guardian:
- Структура модуля: ✅
- Naming conventions: ✅
- TypeScript strictness: ✅

🤖 Sentry Guardian: (если Sentry подключён)
- Критические пути покрыты трейсингом: ✅
- Ошибки захватываются через captureException: ✅

Фича готова к PR.
```

### Подготовка PR

**Нейросеть помогает составить:**
```markdown
## Summary
- [Краткое описание фичи]
- [Ключевые изменения]

## Changes
- Добавлен модуль X
- Изменена логика Y
- Обновлены типы Z

## Testing
- [ ] Ручное тестирование пройдено
- [ ] Сборка успешна (`npm run build`)
- [ ] Линтер пройден (`npm run lint`)
- [ ] 🤖 Агенты: все проверки пройдены

## Screenshots (если UI)
[скриншоты]
```

### Merge
- Создать PR в main
- После апрува — merge
- Удалить feature-ветку

## Справочник агентов

| Агент | Задача | Когда вызывается |
|-------|--------|------------------|
| **🤖 Pragmatic Architect** | Проверка на over-engineering | Фаза 1, Шаг 2.1 (планирование) |
| **🤖 Cache Guardian** | Валидация actions/data fetching | Шаг 2.3 (после написания кода) |
| **🤖 Clean Code Guardian** | Структура, naming, TS | Шаг 2.3 (компоненты > 50 строк) |
| **🤖 Sentry Guardian** | Трейсинг и мониторинг ошибок | Фаза 3 (финальная проверка), при явном запросе |

Документация агентов: `.claude/agents/`

## Чек-лист для нейросети

Перед каждым действием проверять:

- [ ] Прочитан ли `src/docs/<module>.md` если задача касается модуля?
- [ ] Соответствует ли решение паттернам из `CLAUDE.md`?
- [ ] Используются ли существующие компоненты из `src/components/`?
- [ ] Нет ли over-engineering?
- [ ] Проходит ли сборка (`npm run build`)?
- [ ] Проходит ли линтер (`npm run lint`)?
- [ ] Вызваны ли агенты после написания кода?
- [ ] Обновлён ли `src/docs/<module>.md` после изменений?

---

## Чек-лист для разработчика

Перед каждым этапом:

- [ ] Понятна ли цель этапа?
- [ ] Согласован ли план реализации?
- [ ] Есть ли чёткие критерии готовности?

После каждого этапа:

- [ ] Пройдено ли ручное тестирование?
- [ ] Исправлены ли замечания ревью?
- [ ] Пройдены ли проверки агентов?
- [ ] Сделан ли коммит?

---

## Антипаттерны (чего избегать)

1. **Пропуск планирования** — сразу кодить без плана
2. **Игнорирование агентов** — не исправлять замечания Cache/Clean Code Guardian
3. **Игнорирование ревью** — не исправлять замечания
4. **Большие этапы** — этап должен быть обозримым (1-4 часа работы)
5. **Отсутствие тестирования** — "потом проверю"
6. **Коммиты без смысла** — "fix", "update", "changes"
7. **Отклонение от паттернов** — изобретать своё вместо использования существующего
8. **Накопление долга** — откладывать исправления на потом
9. **Over-engineering** — игнорировать замечания Pragmatic Architect
