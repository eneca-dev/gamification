# alarms

Система упреждающих предупреждений о необходимости сменить метку прогресса задачи до прохождения чекпоинта бюджета. Только напоминания — без уведомлений о уже случившихся событиях.

## Логика работы

Алармы рассчитываются в `generate-alarms.ts` (VPS) после основного цикла геймификации (Step 7 в `compute-gamification.ts`). Отображаются баннером на дашборде под стрик-панелью.

Каждый прогон:
1. Авто-резолвит старые нерешённые алармы за прошлые дни (`is_resolved = true`)
2. Удаляет нерешённые алармы за текущую дату
3. Пересчитывает и вставляет актуальные алармы за текущую дату
4. Решённые алармы (вручную или авто) остаются в истории

Два потока:
- **Исполнитель (assignee L3)**: `label_change_soon` — пора сменить метку своей задачи
- **Ответственный за L2**: `team_label_change_soon` — задача L3 в его секции приближается к чекпоинту

Порог: 5% до следующего чекпоинта (20/40/60/80/100%). Если метка уже менялась с последнего чекпоинта — аларм не создаётся.

## Жизненный цикл аларма

1. Скрипт обнаруживает задачу в зоне [checkpoint - 5%, checkpoint) без смены метки → создаёт аларм
2. Юзер видит аларм на дашборде, может отметить чекбоксом "Выполнено" → `is_resolved = true`
3. Следующий прогон: старые нерешённые авто-резолвятся, условие проверяется заново
4. Если юзер сменил метку → аларм не пересоздаётся (условие не выполняется)
5. Если не сменил → новый аларм за сегодня

## Зависимости

- Таблица `alarms` (Supabase, `user_id` = `ws_users.id`)
- VPS-скрипт `generate-alarms.ts`
- Таблицы: `ws_users`, `ws_tasks_l3`, `ws_tasks_l2`, `ws_task_actual_hours`, `ws_task_budget_checkpoints`, `ws_task_percent_snapshots`

## Типы

- `AlarmType`: `label_change_soon` | `team_label_change_soon`
- `AlarmSeverity`: `info` | `warning` | `critical`
- `Alarm`: строка таблицы `alarms`

## Actions

- `resolveAlarm(alarmId)` — отмечает аларм как решённый. Использует `wsUserId` для авторизации. Revalidate: `/`

## Queries

- `getActiveAlarms(wsUserId)` — нерешённые алармы за сегодня. Параметр — `ws_users.id`, не `auth.users.id`

## UI

- Компонент: `src/modules/alarms/components/AlarmsBanner.tsx` (клиентский)
- Расположение: дашборд, под стрик-панелью
- Каждый аларм: иконка severity, title, description, ссылка на задачу в WS, чекбокс resolve
- Сворачивание при > 3 алармов
- Optimistic update при resolve

## Ограничения

- Расчёт и синк данных из WS — 1 раз в сутки, нет realtime
- Внутри дня аларм убирается только вручную (чекбокс)
- Решённый вручную аларм может появиться снова на следующий день, если условие сохраняется
- RLS: маппинг `auth.uid()` → `ws_users.id` через `profiles.email`
