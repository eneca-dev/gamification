# Методика: дашборд внедрения геймификации (/admin/adoption)

Страница пересчитывается при каждом открытии: сервер делает свежие запросы к Supabase,
данные актуальны на момент последнего ночного прогона VPS-скриптов.

**Запуск геймификации:** 1 июля 2026
**Когорта:** проектировщики — активные сотрудники отделов с `group_type = 'designer'`
из `admin_department_groups` (19 отделов, ~377 человек). Все метрики считаются только по когорте.
**Период ДО:** 29–30 июня 2026 (два рабочих дня, пн–вт, перед запуском)
**Период ПОСЛЕ:** 1 июля 2026 – вчерашний день (сегодняшние данные ещё не синхронизированы)
**Верхняя граница дат во всех запросах:** `getYesterday()` — UTC-дата вчера

Структура дашборда: 4 блока — Общая картина, Дисциплина Worksection, Revit-плагины,
Активность в системе.

---

## Общие правила

**Когорта в SQL** — хелперы `adoption_designer_ids()` (ws_users.id) и
`adoption_designer_emails()` (lower(email)): активные `ws_users`, чей `department`
входит в `admin_department_groups` с `group_type = 'designer'`.

**Ключи пользователей.** Один человек живёт в трёх системах идентификаторов:
- `ws_users.id` — стрики, балансы, вердикты, отчёты, `gratitudes.sender_id/recipient_id`, `shop_orders.user_id`
- `profiles.user_id` (auth uid) — `chat_messages.user_id`; с ws_users не пересекается
- `lower(email)` — elk-таблицы плагинов; связь profiles ↔ ws_users тоже только по email

**Рабочие дни** — будни минус праздники (`calendar_holidays`) плюс рабочие субботы-переносы
(`calendar_workdays`). Метрики плагинов и вердиктов считаются только по рабочим дням.

**Вердикты** — UNION двух таблиц: `ws_daily_statuses` (живые, с 01.07) и
`ws_daily_statuses_baseline` (ретроспективный бэкфил за 29–30.06, миграция 086, по тем же
правилам VPS). Отсутствующие исключаются: знаменатель = green + red.

**Пагинация** — таблицы с > 1 000 строк вычитываются постранично (PAGE = 1 000)
с обязательным `.order('id')`.

## Блок 1. Общая картина

SQL (все функции — миграция 087): `get_adoption_coverage()`, `get_adoption_earned`,
`get_adoption_users_daily`, `get_adoption_improved_daily_split`,
`get_adoption_improved_prelaunch`, `get_adoption_revit_daily`; ряд зелёных дней
переиспользуется из блока 2 (`get_adoption_ws_daily`).

### Карточки охвата

| Карточка | Формула |
|---|---|
| Проектировщиков в когорте | `COUNT(adoption_designer_ids())` · «из N сотрудников» = COUNT активных `ws_users` |
| Авторизовались | профили с email из когорты |
| Заработано кристаллов с 01.07 | `get_adoption_earned()`: SUM(coins > 0) по `gamification_transactions` когорты; сплит «% вошедшими» — доля суммы у тех, кто есть в profiles (по email) |

### График «Вход в систему и личные улучшения» (календарные дни с 29.06)

```
logged_in(день)      = |{профили когорты с created_at::date <= день}|  (накопительно)
improved_ws(день)    = |{u: доля зелёных u с 01.07 по день > доля зелёных u за 29–30.06}|
improved_revit(день) = |{u: запусков/раб.день с 01.07 по день > запусков/раб.день за 29–30.06}|
```

Каждый сравнивается сам с собой: линии улучшений показывают, сколько человек на дату D
работают лучше собственного уровня ДО. У improved_ws в базе только те, у кого были вердикты
29–30.06 (~349 чел.); improved_revit включает тех, кто до запуска плагинами не пользовался
(их baseline = 0). Точки 29–30.06 — естественный фон улучшений без геймификации:
`get_adoption_improved_prelaunch` считает то же сравнение, но относительно
25–26.06 (пред. рабочая пара дней); вердикты за 25–26.06 вычисляются на лету по логике
бэкфила (миграция 086). Фон: ~78 чел. WS / ~63 чел. Revit к 30.06 — недельные колебания дают
«улучшившихся» и без геймификации (regression to the mean), сигнал — превышение этого фона.

Две вкладки. «Вся выборка» — исходный вид: вход в систему + суммарные линии улучшений.
«Вошли vs не вошли» — те же улучшения раздельно по факту входа в приложение: сплошные —
вошедшие, пунктирные — не вошедшие (квази-контроль); шкала и серые горизонтали — размеры
групп. SQL: `get_adoption_improved_daily_split`, split по наличию email
в profiles на сегодня (не на дату D); сумма групп равна суммарной линии (проверено по дням).

### График «Зелёные дни» — ряд `green_pct` из `get_adoption_ws_daily` (с 29.06, включая бэкфил)

### График «Revit-плагины»

```
users(день) = COUNT(DISTINCT lower(user_email)) по когорте за рабочий день
```

До 30.06 — `elk_plugin_launches_baseline`, с 01.07 — `elk_plugin_launches` (периоды не пересекаются).

### Скрытый список «Вход по отделам и командам»

Кнопка «Показать вход по отделам и командам» под карточками охвата: список разбит на
две независимо скроллируемые колонки (первая половина отсортированного списка отделов —
в первую колонку, продолжение — во вторую), поэтому раскрытие в одной колонке не сдвигает
другую. Вложенность: отдел → команда → человек. Отдел с одной командой раскрывается сразу
в список людей (промежуточная команда пропускается). Сортировка по возрастанию % входа
(худшие сверху) на уровне отделов и команд; не вошедшие — первыми в списке людей.
Цвет: < 50% — красный, ≥ 50% — зелёный (то же на человеке — цветной маркер и подпись).

SQL: `get_adoption_login_by_department()` — по каждой паре (отдел, команда)
из `ws_users` считает `total` и `logged_in` (email есть в `profiles`); в TS команды
сворачиваются в отдел (сумма total/logged_in). `get_adoption_login_users()` —
пофамильный список с тем же условием входа, ФИО из `ws_users`; в TS раскладывается по ключу
(отдел, команда) в списки `users` каждой команды.

## Блок 2. Дисциплина Worksection

Итоговый вердикт + два главных нарушения. Нарушения — доли от отслеживаемых,
«чем меньше — тем лучше».

### Карточки ДО → ПОСЛЕ → Δ

Агрегаты взвешены по людям (сумма числителей / сумма tracked за период), а не средние
дневных процентов — дни с меньшим составом не искажают итог.

```
«Зелёные» дни        = SUM(green) / SUM(tracked)                    ДО ~47% → ПОСЛЕ ~56%
Отчёт в задачу в статусе не «В работе» = SUM(wrong_status_report) / SUM(tracked)  ДО ~26.4% → ПОСЛЕ ~17.3%
Нет отчёта           = SUM(tracked − reported) / SUM(tracked)       ДО ~26.5% → ПОСЛЕ ~27.1%
```

Счётчики причин — `get_adoption_red_reasons_daily(p_from, p_to)`: разбор jsonb `red_reasons`
красных вердиктов по дням; `wrong_status_report` = часы в задачу не в статусе «В работе»
(правило 4 WS, статус на дату — из `ws_task_status_changes`), `red_day` = нет отчёта в тот же
день и совпадает с `tracked − reported` из `get_adoption_ws_daily`
(в `ws_daily_reports` попадают только отчёты, сданные в течение 1 дня).
**Важно:** в живых вердиктах бывает несколько записей `wrong_status_report`
за день (по записи на задачу), в бэкфиле — одна; счётчики считают `COUNT(DISTINCT user_id)` —
людей, а не записи (иначе ПОСЛЕ-доля была бы завышена — 22.5% вместо 17.3%).
В подсказках карточек нарушений — абсолюты: среднее число случаев на рабочий день периода.

### Скрытые списки нарушителей

Кнопка «Показать списки нарушителей» под карточками: два скроллируемых списка за период
ПОСЛЕ — у кого были красные дни по каждой причине, с числом дней (по убыванию).
SQL: `get_adoption_red_users(p_from, p_to)` — ФИО + отдел из
`ws_users`, `days = COUNT(DISTINCT date)`.

**Исключена команда «Декретный» — только из этого списка.** Причина:
decree-сотрудники почти никогда не попадают в `ws_user_absences` (HR фиксирует в WS
отпуск/больничный, но не декрет), поэтому получают `red_day` каждый рабочий день просто
за отсутствие отчёта — это пробел в источнике данных WS, а не реальное нарушение.
Проверено: 15 из 16 decree-сотрудников выборки — вообще без единой записи в
`ws_user_absences`; после исключения список «Нет отчёта» короче на 14 человек (202 → 188).
Карточки, %, график и остальные метрики блока decree-команду **не исключают** — эта
фильтрация есть только в `get_adoption_red_users`.

### График «Зелёные дни и нарушения по дням»

Те же три метрики, что и в карточках, по дням с вердиктами (= рабочие дни):
`green_pct` из `get_adoption_ws_daily` (сплошная линия), `wrong_task_pct` и `no_report_pct`
(пунктирные, «чем ниже — тем лучше») — считаются в TS из дневных счётчиков причин
и tracked того же дня.

### Эффект вовлечения (квази-контрольная группа) — главный инсайт

SQL: `get_adoption_login_effect(p_to)`. Когорта делится по факту входа в приложение
(profiles по email):

```
Вошли (203 чел.):    зелёных 58.6% ДО → 74.3% ПОСЛЕ  (+15.7 пп)
Не вошли (171 чел.): зелёных 33.0% ДО → 31.5% ПОСЛЕ  (−1.5 пп)
```

Правила WS одинаковы для всех, но рост — только у вошедших: эффект нельзя объяснить сезоном
или общим давлением руководства. Оговорка: группы самоотобранные (вошли более мотивированные),
но отсутствие роста в контрольной группе исключает внешние причины.

## Блок 3. Revit-плагины: интенсивность использования

Только рабочие дни, только когорта (фильтр по `adoption_designer_emails()` на стороне TS).
Данные: `elk_plugin_launches_baseline` с 04.05 (историческая выгрузка) + `elk_plugin_launches`
с 01.07. Окно ДО для карточек — **июнь** (стабильная месячная база), ПОСЛЕ — с 01.07;
все метрики нормированы на рабочий день, поэтому окна разной длины сопоставимы.

```
Активных в средний рабочий день = AVG(|уникальных email за раб. день|)      79 → 104 (+32%)
Запусков за рабочий день        = AVG(SUM(launch_count) за раб. день)       312 → 388 (+24%)
Дней использования в неделю     = AVG по полным неделям(user_days / users)  2.6 → 3.2
```

Полная неделя = последний рабочий день недели ≤ вчера; неделя запуска (пн 29.06) смешивает
ДО и ПОСЛЕ и не входит ни в одно окно «дней в неделю».

График «Активность по дням» (рабочие дни с 29.06): активные пользователи (левая ось)
и запуски (правая ось) за каждый день. Вертикаль — запуск 1 июля.

**Почему интенсивность, а не охват:** охват плоский и ростом не является. Недельная
аудитория стабильна ~150–155 чел. девять недель до запуска и после (170 в пиковую неделю);
месячный охват июня (213) даже выше охвата с 01.07 (203). Прежние карточки «Пользовались
хоть раз 107 → 203» (окно 2 дня против 15) и «Начали после запуска 101» (честное месячное
окно даёт 21) удалены как артефакты окон. Рост реален в частоте: те же люди пользуются
плагинами чаще — это согласуется с квази-контролем (вошедшие 24% → 40%).

### Эффект вовлечения (квази-контроль, аналог блока 2)

Выборка делится по факту входа в приложение (profiles по email). Метрика — доля группы,
запускавшая плагины в средний рабочий день (нормировка по числу рабочих дней делает
периоды разной длины сопоставимыми):

```
active_pct = |пар (день, email) группы| / раб.дней периода / размер группы × 100
Вошли:    ~24% ДО → ~40% ПОСЛЕ
Не вошли: ~16% ДО → ~13% ПОСЛЕ
```

Считается на стороне TS в `getAdoptionPlugins` из тех же elk-строк, что и остальные карточки.

Дневной график использования — в блоке 1.

## Блок 4. Активность когорты в системе (с 1 июля)

Единая сетка карточек. Кристаллы — RPC `get_adoption_crystal_stats(p_from)`,
остальное — абсолютные счётчики с фильтром по когорте на стороне TS
(ключи — см. «Общие правила»):

| Карточка | Источник | Формула |
|---|---|---|
| Зарабатывают кристаллы | `gamification_transactions` | COUNT(DISTINCT user_id), coins > 0, created_at ≥ 01.07, user_id ∈ ws-ids · % от когорты |
| Кристаллов на балансах | `gamification_balances` | SUM(total_coins) по когорте · среднее = сумма / зарабатывающих |
| Потрачено кристаллов | `gamification_transactions` | −SUM(coins), coins < 0, created_at ≥ 01.07 |
| Благодарности | `gratitudes` | sender_id ∈ ws-ids · уникальные отправители / получатели |
| Магазин | `shop_orders` (status ≠ cancelled) | user_id ∈ ws-ids · уникальные покупатели |
| Чат-бот | `chat_messages` (role = user) | user_id ∈ profile-ids когорты · уникальные пользователи |

## Методологические ограничения

1. **Данные по вчерашний день** — верхняя граница `getYesterday()`.

2. **Период ДО — всего два дня.** Точечная оценка, а не средняя за сезон:
   чувствительна к разовым событиям 29–30 июня. Зато нет сезонного смещения май-против-июля.

3. **Бэкфил 29–30.06 — ретроспективный расчёт**, не живые вердикты VPS. Логика та же,
   но проверки, требующие созревания (динамика задач), в бэкфиле отсутствуют —
   красных за эти дни может быть чуть меньше, чем насчитал бы VPS.

4. **Эффект новизны.** Первые недели после запуска дают всплеск вовлечённости.

5. **Причинность не доказана строго**, но сравнение вошедших/не вошедших — сильный аргумент:
   у не вошедших дисциплина не изменилась.

6. **3 июля — праздник в Беларуси**, исключён из рабочих дней (провал на дневных графиках —
   это выходной, не падение).
